Crypto Pigeon — E2EE Messaging Application Restructure (Final v3)

Scope Statement

Crypto Pigeon uses libsignal-based end-to-end encryption with PQXDH-compatible session establishment. It is an independent implementation and has not been audited or endorsed by Signal.

This plan does not claim Signal-equivalent privacy, complete metadata anonymity, protection against endpoint compromise, unbreakable encryption, or perfect forward secrecy against all adversary classes.

The relay processes network and routing metadata during active connections but is designed not to persist it in application logs. Infrastructure-level logging by reverse proxies, hosting providers, and operating systems may still occur and must be separately configured.

Device Model

This MVP is strictly single-device per user account. One active device per user. Activating a replacement device revokes the previous device. Contacts receive an identity-change warning. No historical messages are transferred. No message fanout, no linked-device key verification, no per-device delivery accounting.

True multi-device support would require separate Signal sessions per recipient device, one encrypted envelope per device, separate prekey bundles per device, attachment download accounting per device, delivery state aggregation, and linked-device key verification. None of that is in scope.

Post-Quantum Ratchet Status

The exact post-quantum ratchet behavior of libsignal 0.96.4 will be determined through inspection of the pinned API, upstream implementation, and interoperability tests. The SignalMessage structure in 0.96.4 contains a pqRatchet field, but the public TypeScript API does not expose obvious exports named SPQR or Triple Ratchet. Crypto Pigeon will not claim continuous post-quantum security unless that behavior is verified. Version pinning is maintained because Signal explicitly states outside use is unsupported and APIs may change.

Key Separation Architecture

Four distinct key domains. No key is ever reused across domains.

┌─────────────────────────────────────────────────────┐
│  1. VAULT MASTER KEY HIERARCHY                      │
│     password (≥12 chars, reject common/breached)    │
│       ↓ Argon2id(passphrase, argon2_salt,           │
│         memoryCost=65536KiB†, timeCost=3†,          │
│         parallelism=1†)                             │
│     Key Encryption Key (KEK, 256-bit)               │
│       ↓ AES-256-GCM-unwrap(KEK, wrapped_vmk,       │
│         nonce, aad="vmk-wrap-v1")                   │
│     Vault Master Key (VMK, 256-bit random)          │
│       ↓ HKDF(VMK, hkdf_salt‡, context)             │
│     ├── sqlcipher_key        "sqlcipher-key-v1"     │
│     ├── field_encryption_key "field-aead-v1"        │
│     ├── attachment_meta_key  "attachment-meta-v1"   │
│     └── local_token_key     "local-token-v1"        │
│                                                     │
│  † Baseline minimum. Calibrate at first setup for   │
│    approximately 300–750ms on the machine.           │
│  ‡ hkdf_salt is a separate random salt, NOT the     │
│    same as the Argon2 password salt.                 │
│  Password change: re-wrap VMK with new KEK.         │
│  No bulk re-encryption of records required.          │
├─────────────────────────────────────────────────────┤
│  2. SIGNAL IDENTITY KEYS (libsignal only)           │
│     libsignal IdentityKeyPair                       │
│     Serialized/deserialized only through libsignal. │

│     raw X25519/Ed25519. Never converted into        │
│     device-auth keys.                               │
│     + signed prekey pairs                           │
│     + one-time prekeys (classical)                  │
│     + Kyber prekeys (ML-KEM, post-quantum)          │
│                                                     │
│  NEVER used for: relay auth, API auth, admin,       │

├─────────────────────────────────────────────────────┤
│  3. DEVICE AUTH SIGNING KEY (relay authentication)  │

Relay Envelope Format

interface RelayEnvelopeV1 {
  version: 1;
  envelopeId: string;     // 128-bit cryptographically random (NOT UUIDv7)
  mailboxId: string;      // opaque recipient routing capability
  ciphertext: Uint8Array; // opaque sealed payload — indivisible bytes
}

Relay applies its own queue TTL from server receipt time. No client-supplied expiry in the outer envelope. All message metadata (sender, type, timer, filename, etc.) inside E2EE payload only.

Key Version Tracking (Operational, Not Security Authority)

CREATE TABLE device_key_versions (
  device_id   UUID NOT NULL REFERENCES device_identity_keys(device_id),
  key_type    TEXT NOT NULL,    -- 'identity', 'signed_prekey'

IMPORTANT: The device_key_versions table is retained for operational history but is NOT a security authority. A malicious relay controls that table and can replace both the identity key and version information. Authoritative identity-change detection happens exclusively in the local daemon via TOFU pinning (see Identity Pinning section below).

Bundle Consumption (POST /api/prekeys/bundle)

POST, not GET — this operation mutates state by consuming one-time prekeys. GET would allow caches, crawlers, and retries to consume prekeys.

BEGIN;
  -- Lock and consume one classical one-time prekey
  UPDATE one_time_prekeys
    SET consumed = true, consumed_at = now()
    WHERE id = (
      SELECT id FROM one_time_prekeys
      WHERE device_id = $1 AND consumed = false
      ORDER BY uploaded_at ASC
      LIMIT 1
      FOR UPDATE SKIP LOCKED
    )
    RETURNING signal_prekey_id, public_key;

  -- Lock and consume one PQ one-time prekey
  UPDATE pq_one_time_prekeys
    SET consumed = true, consumed_at = now()
    WHERE id = (
      SELECT id FROM pq_one_time_prekeys
      WHERE device_id = $1 AND consumed = false AND prekey_type = 'one_time'
      ORDER BY uploaded_at ASC
      LIMIT 1
      FOR UPDATE SKIP LOCKED
    )
    RETURNING signal_prekey_id, public_key, signature;
COMMIT;

Fallback when inventory empty:

- Classical one-time prekey: may be absent (libsignal PreKeyBundle allows preKeyId to be null)
- PQ one-time prekey: fall back to signed last-resort Kyber prekey (prekey_type = 'last_resort'), which is NOT consumed
- Trigger immediate replenishment warning to the device owner (via WebSocket notification or next poll)

Identity Pinning (Client-Side TOFU)

Identity-change detection is authoritative only in the local daemon, not the relay.

First contact with User B:
  → fetch bundle from relay
  → store User B's identity public key locally (Trust On First Use)
  → display safety number for optional out-of-band verification

Subsequent bundle fetches for User B:
  → compare returned identity key against locally pinned key
  → IF different:
      → reject automatic session replacement
      → mark contact as identity_changed
      → display BLOCKING warning in UI
      → require explicit user approval before proceeding
      → update safety number display

Limitation:
  A first-contact relay substitution (MITM at initial bundle fetch)
  cannot be detected without out-of-band safety-number verification.
  This is inherent to TOFU and is documented as a known limitation.

Localhost Bootstrap Authentication

Single origin. UI and API from same http://127.0.0.1:<random-port>/.

Startup Flow

1. Daemon binds to 127.0.0.1 on random available port
2. Generate one-time 256-bit bootstrap secret
3. Terminal prints: "Open: http://127.0.0.1:PORT/bootstrap#SECRET"
4. Browser JS reads fragment, POSTs { secret } to /api/bootstrap
5. Server verifies, atomically invalidates bootstrap secret
6. Server sets HttpOnly, SameSite=Strict, Path=/ session cookie
7. Server returns CSRF token in response body (JS memory only)
8. Bootstrap secret permanently expired

CSRF Token Refresh After Page Reload

After page reload, the in-memory CSRF token is lost. Provide:

GET /api/session
  Requires: valid session cookie
  Returns: { csrfToken, vaultState }
  Purpose: re-establish CSRF token after page reload

Session Invalidation

The local session cookie and all associated state are invalidated when:

- Vault locks (inactivity timeout or explicit lock)
- Vault unlocks again (new session required)
- Daemon restarts
- User explicitly logs out

This prevents a browser tab holding an old cookie from regaining access after vault re-open. Each unlock cycle generates a new session generation/version number; the session cookie is bound to that generation.

WebSocket Handshake Validation

Every WebSocket upgrade must validate:

1. Session cookie (valid and current generation)
2. Origin header (http://127.0.0.1:PORT only)
3. Host header (127.0.0.1:PORT only)
4. Vault unlocked state
5. Session generation matches

Attachment Encryption (Deterministic Design)

Nonce Strategy (Fixed Decision)

Random 12-byte nonce per chunk. Stored alongside each encrypted chunk. At 25MB limit with 64KB chunks (~400 chunks max), random 96-bit nonces have negligible collision probability under a single attachment key.

Per-Attachment Flow

1. Generate random attachment key: crypto.randomBytes(32) → AES-256-GCM
2. Chunk plaintext into 64KB blocks
3. For each chunk:
   - Generate random 12-byte nonce: crypto.randomBytes(12)
   - AEAD additional data: attachmentId || chunkIndex || totalChunkCount || protocolVersion
   - Encrypt chunk with AES-256-GCM(attachment_key, nonce, plaintext, aad)
   - Store: { encryptedData, nonce, chunkIndex }
4. Compute full-file SHA-256 digest over complete plaintext
5. Upload encrypted chunks to relay with high-entropy blob ID (128-bit random)
6. Finalize upload transaction on relay

Authenticated Manifest (Inside Signal E2EE Payload)

interface AttachmentManifest {
  attachmentId: string;          // 128-bit random
  blobId: string;                // relay blob capability
  originalFilename: string;      // sanitized
  claimedMimeType: string;       // NOT trusted for rendering
  totalPlaintextSize: number;
  chunkCount: number;
  chunkSize: number;             // 65536
  chunkNonces: string[];         // base64-encoded per-chunk nonces
  fullFileDigest: string;        // SHA-256 hex of full plaintext
  attachmentKey: string;         // base64 AES-256-GCM key
  encryptionVersion: 1;
}

Download Verification

1. Receive manifest from decrypted Signal message
2. Download encrypted chunks using blobId (requires session auth)
3. Decrypt each chunk with manifest's attachment key + stored nonce
4. Verify AEAD tag (implicit in GCM decrypt)
5. Verify chunk count matches manifest
6. Reject duplicate or reordered chunk indices
7. Reassemble plaintext
8. Compute SHA-256 over reassembled plaintext
9. Compare against manifest digest using timing-safe comparison
10. Reject on any mismatch
11. Sanitize filename: strip path separators, control chars, null bytes, limit length
12. Render using blob: URLs; never trust claimedMimeType for execution

Single-Device Blob Deletion

download all chunks
  → decrypt each chunk
  → verify AEAD tags
  → verify full-file digest (timing-safe)
  → store decrypted attachment durably in vault
  → send blob acknowledgment to relay
  → relay deletes blob

One device per user means no multi-device download accounting needed.

Disappearing Messages

Property: Disappearing messages perform best-effort logical deletion from compliant Crypto Pigeon clients. They do not prevent screenshots, screen recording, text copying, client modification, database backup, or physical data recovery from storage media.

Semantics

- Sender copy timer starts relative to send time
- Recipient copy timer starts relative to first display/read
- Timer changes are E2EE control messages (inside encrypted payload)
- Relay queue TTL is independent (server receipt time)
- Attachment deletion follows local message deletion
- Read receipts are optional and E2EE (not relay metadata)

Timer Values

off | 30s | 5m | 1h | 1d | 1w

Timer Updates

A timer update is an encrypted conversation-control message. Each compliant client applies the latest valid timer update from the point it is received. Neither party can cryptographically force deletion on the other endpoint.

Use a monotonically increasing conversation-control sequence number to prevent old timer updates from overriding newer ones. If a received sequence number is ≤ the current local sequence number, the update is ignored.

Delivery Acknowledgment Sequence

Ack happens only after durable local storage.

receive envelope over WebSocket
  → validate envelope bounds and version
  → deduplicate against local seen_envelope_ids
  → decrypt ciphertext via Signal adapter
  → BEGIN TRANSACTION (SQLite/SQLCipher WAL mode, PRAGMA synchronous = FULL)
      → store decrypted message in vault
      → store updated Signal session/ratchet state
      → store envelope ID in deduplication table
    COMMIT
  → send delivery acknowledgment to relay
  → relay deletes queued envelope

No per-message WAL checkpoint. PRAGMA synchronous = FULL ensures durability at transaction commit. Periodic checkpoints occur naturally or can be triggered at intervals (e.g., every 60 seconds).

Delivery States

State              Meaning
relay_accepted     Relay received and queued the envelope
delivered          Recipient vault durably stored the decrypted message
displayed          Recipient UI rendered the message (starts disappear timer)

Read receipts are encrypted messages, not relay metadata.

Activation Code Design

Properties

- 128 bits of entropy: crypto.randomBytes(16) → base32 encoding
- Expiration: 10 minutes (configurable)
- Single use: atomically consumed
- Attempt limits: 5 attempts per request ID, then locked
- Bound to device: activation binds code to submitted device public keys
- Server-side storage: HMAC-SHA256(pepper[version], activation_code)
  - Server pepper loaded from environment
  - Pepper version stored alongside HMAC: pepper_version INTEGER NOT NULL
  - Current and previous pepper retained until all codes under previous version have expired
  - With 128-bit random codes and 10-minute lifetime, frequent pepper rotation provides little practical benefit; stable deployment secret with planned operational rotation is sufficient
- No plaintext storage, logs, URLs
- CLI history warning: administrator informed that code appears in terminal output

Vault Key Hierarchy

User passphrase (≥12 chars, reject common/breached)
    │
    ▼ Argon2id(passphrase, argon2_salt,
    │          memoryCost†, timeCost†, parallelism†)
    │
Key Encryption Key (KEK, 256-bit)
    │
    ▼ AES-256-GCM-unwrap(KEK, wrapped_vmk, nonce, aad="vmk-wrap-v1")
    │
Vault Master Key (VMK, 256-bit random, generated once)
    │  (separate hkdf_salt, NOT argon2_salt)
    │
    ├─ HKDF(VMK, hkdf_salt, "sqlcipher-key-v1")    → SQLCipher key
    ├─ HKDF(VMK, hkdf_salt, "field-aead-v1")       → field AEAD key
    ├─ HKDF(VMK, hkdf_salt, "attachment-meta-v1")   → attachment metadata key
    └─ HKDF(VMK, hkdf_salt, "local-token-v1")       → local session token key

Password Policy

- Minimum 12 characters
- Reject common or breached passwords (bundled top-10k list check)
- Encourage password-manager-generated passphrases in UI
- Calibrate Argon2id at first vault creation for approximately 300–750ms on the machine
- 64 MiB memory, 3 iterations, parallelism 1 as minimum baseline, not absolute
- Separate random HKDF derivation salt from Argon2 password salt

Platform Security

- Windows: icacls on vault directory, restrict to current user

Metadata Privacy (Honest)

Relay sees during active connections

Attachment filename, MIME type      Inside E2EE payload
Disappearing timer values           Inside E2EE payload
Private keys                        Never leave daemon

Application does NOT persist

- IP addresses not stored in database or logs
- HTTP/WebSocket access logs disabled
- No analytics, telemetry, fingerprinting, third-party scripts

Honest caveats

- Infrastructure may log connection metadata separately
- Tor is optional, not mandatory; does not constitute anonymity guarantee
- Sealed-sender delivery not implemented in this MVP
- Traffic analysis possible for network-level adversary
- Sender authenticates to WebSocket before sending — sender info in ciphertext alone does not achieve sealed-sender

Canary-Based Verification (Replaces WAL Inspection Claims)

Insert unique canary plaintext strings into messages. After relay processing, scan:

- Relay database columns (SELECT across all tables)
- Relay application logs
- Docker container logs
- Temporary files in relay working directory
- Error/crash output captured during test

This does not prove absolute absence of plaintext leakage but is a meaningful, reproducible test.

Known Limitations (Pre-documented)

1. Not Signal-equivalent — independent, unaudited implementation
2. No sealed-sender delivery
3. No continuous post-quantum forward secrecy (unless libsignal 0.96.4 provides it)
4. TOFU vulnerable to first-contact MITM without OOB verification
5. Metadata visible to relay during connections
6. Single-device only — no multi-device
7. Disappearing messages are best-effort logical deletion
8. AGPL-3.0 licensing obligations apply
